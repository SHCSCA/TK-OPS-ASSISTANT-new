## 二、产品概述

### 2.1 产品定义

TK-OPS 是一款面向 TikTok Shop 团队的企业级桌面应用。
它服务于账号管理、内容生产、数据分析、自动化运营、客户关系管理和 AI 工作流。
产品不是单个工具集合，而是一套统一桌面工作台。
用户进入系统后，可以在同一壳层中完成选题、分析、生成、剪辑、发布、互动、客户跟进与复盘。

### 2.2 产品要解决的核心问题

- 现有流程分散在网页工具、本地脚本、表格与聊天工具中。
- 多角色之间缺少统一上下文。
- 内容生成和执行环节脱节。
- 数据分析结果难以转化为具体动作。
- 多账号、多任务、多素材的状态难以统一追踪。
- AI 供应商、模型和提示词体系零散，维护成本高。
- 后台任务执行缺少队列、调度、日志和重试机制。
- 现有工具往往偏功能堆叠，缺少可扩展架构。

### 2.3 产品定位

- 面向 TikTok Shop 运营团队的桌面操作系统。
- 面向多角色协作的统一工作台。
- 面向 AI 原生内容生产与自动化执行的平台。
- 面向后续插件扩展的企业级桌面壳层。

### 2.4 核心价值

- 一体化，减少工具切换。
- 可视化，让状态一眼可见。
- AI 原生，让生成与分析变成系统能力。
- 可治理，让配置、权限、日志、主题和模型统一管理。
- 可扩展，让新页面、新能力、新供应商可接入同一框架。

## 三、目标用户

### 3.1 运营经理

- 负责账号矩阵、团队执行、客户价值和经营结果。
- 需要全局看板。
- 需要客户与任务统一追踪。
- 需要快速发现异常并分派动作。

### 3.2 内容创作者

- 负责标题、脚本、文案、素材、剪辑和创意试验。
- 需要高频使用 AI 生成。
- 需要生成后继续编辑和加工。
- 需要批量生产和多版本对比。

### 3.3 数据分析师

- 负责竞品、蓝海、利润、互动、报表与可视化分析。
- 需要跨数据源工作。
- 需要把洞察快速反馈给内容和运营团队。

### 3.4 自动化运营专员

- 负责采集、调度、自动回复、定时发布和异常重试。
- 需要稳定的任务系统。
- 需要清晰的日志与运行状态。

### 3.5 系统管理员

- 负责 AI 供应商、权限、网络、日志、版本、密钥与系统健康。
- 需要统一配置界面和可观测性能力。

### 3.6 次级用户

- 中小工作室负责人
- MCN 团队负责人
- 客服人员
- 客户运营人员
- 海外店铺运营人员

## 四、产品愿景

### 4.1 长期愿景

TK-OPS 要成为 TikTok Shop 团队的标准化桌面操作平台。
它要把内容生产、经营分析、自动执行、客户跟进和 AI 协同纳入同一个可持续演进的系统。

### 4.2 中期愿景

- 建立稳定的 plugin shell。
- 建立统一 AI 接入层。
- 建立角色专区工作台。
- 建立本地优先的数据与任务治理体系。

### 4.3 短期愿景

- 覆盖全部 26 个页面。
- 打通首次启动、日常运营、AI 内容生产、任务调度和复盘。
- 落地浅色与深色主题。
- 落地 AI 供应商、模型、角色、回退链路的可配置能力。

## 五、竞争优势

### 5.1 相对参考项目的产品优势

- 不是单一工具集合，而是按业务域组织的系统产品。
- 不只聚焦单点 AI 文案，而是覆盖分析、生成、执行、复盘全链路。
- 引入五大专区，降低多角色使用门槛。
- 把内容、数据、自动化、CRM 串成闭环。

### 5.2 相对参考项目的架构优势

- 使用 PySide6 和 Python 3.11+，便于长期维护。
- 使用 Clean Architecture 与 MVVM，边界更清晰。
- 使用 plugin shell，页面与壳层解耦。
- 使用 ConfigBus 管理配置广播。
- 使用 ThemeEngine 管理主题令牌。
- 使用 RouteRegistry 管理所有页面与 stub 路由。
- 使用 LiteLLM 统一多供应商 AI 调用。

### 5.3 相对参考项目的治理优势

- 供应商可切换。
- 模型可切换。
- 角色预设可编辑。
- 任务有队列、有调度、有日志。
- 页面接入统一权限、主题、搜索和状态体系。

## 七、设计系统要求

### 7.1 品牌与主题色

- 主品牌色：`#00F2EA`
- 浅色背景：`#F5F8F8`
- 深色背景：`#0F2323`

### 7.2 专区角色色

- 运营经理专区：`#FF6B6B`
- 内容创作者专区：`#4ECDC4`
- 数据分析师专区：`#95E1D3`
- 自动化运营专区：`#F38181`

### 7.3 主题规则

- 必须支持浅色主题。
- 必须支持深色主题。
- 页面不可直接硬编码完整配色。
- ThemeEngine 必须基于语义令牌渲染界面。
- 关键按钮使用品牌主色。
- 区域导航与域级提示使用专区色加强辨识。

### 7.4 壳层布局规则

- 顶部标题栏提供全局搜索、通知和用户入口。
- 左侧导航提供分组和图标。
- 中部内容区由 `QStackedWidget` 承载。
- 右侧详情抽屉按页声明启用。
- 底部状态栏展示系统状态。

## 九、功能需求，dashboard 域

### 9.1 TK-OPS Dashboard

- 页面定位：全局经营与运行态概览页。
- 主要用户：运营经理、系统管理员、团队负责人。
- 核心目标：帮助用户在短时间内识别当天优先动作。
- 核心模块：全局搜索、KPI 卡片、AI 任务总览、AI 引擎状态卡、任务趋势图、最近活动、快捷入口、异常横幅。
- 关键指标：账号总数、异常账号数、运行任务数、失败任务数、AI 调用量、客户待跟进数。
- 关键操作：跳转账号管理、AI供应商配置、任务队列、任务调度、定时发布、CRM客户关系管理。
- 数据来源：账号域、任务域、AI 域、CRM 域、系统域。
- AI 能力：展示模型成功率、成本趋势、最近失败原因、角色预设使用频次。
- 详情抽屉：点击 KPI 后展示来源、趋势、关联任务和快捷跳转。
- 验收要求：用户打开后可在一屏内看懂当前经营状态。

## 十、功能需求，account 域

### 10.1 账号管理

- 页面定位：多账号治理中心。
- 主要用户：运营经理、运营专员、系统管理员。
- 页内标签：全部账号、在线、离线、异常。
- 核心模块：账号列表、筛选栏、批量导入、详情抽屉、环境配置、风险提醒。
- 关键操作：新增账号、编辑账号、删除账号、导入 CSV、按状态筛选、测试登录、打开隔离环境、处理异常。
- 展示字段：平台、地区、负责人、最近登录、代理、风险等级、状态。
- 数据联动：联动分组管理、设备管理、任务调度、定时发布、CRM。
- AI 能力：提供异常风险说明与处理建议。
- 安全要求：Cookie 与敏感凭据不可明文展示。
- 验收要求：异常账号可以被快速筛出并进入处理流程。

## 十一、功能需求，operations 域

### 11.1 域组说明

- operations 域是正式业务域。
- 首期虽然没有独立 mockup 页面，但必须存在完整导航结构。
- 该域承担运营经理专区的业务承接责任。

### 11.2 运营中心

- 页面定位：运营域 hub 页。
- 主要用户：运营经理。
- 核心模块：经营摘要卡、待办列表、异常区、客户热点区、跳转区。
- 关键操作：跳转 CRM客户关系管理、订单管理、退款处理、客服中心。
- 数据联动：联动 CRM、自动回复、利润分析、任务系统。
- 验收要求：入口清晰、无死链、可承接后续扩展。

### 11.3 订单管理

- 页面定位：订单工作页，首期可为 stub 页。
- 后续能力：订单列表、筛选、导出、客户跳转、利润联动。
- 验收要求：路由存在，可纳入权限体系。

### 11.4 退款处理

- 页面定位：退款工单页，首期可为 stub 页。
- 后续能力：退款工单、证据附件、SLA、责任归属、处理记录。
- 验收要求：路由存在，可纳入日志中心。

### 11.5 客服中心

- 页面定位：客服工作页，首期可为 stub 页。
- 后续能力：历史会话、客户档案、自动回复、人工接管。
- 验收要求：路由存在，可与 CRM 和自动回复控制台联动。

## 十二、功能需求，content 域

### 12.1 视觉编辑器

- 页面定位：视频剪辑与视觉编辑工作台。
- 主要用户：内容创作者、运营专员。
- 页内模式：时间轴、关键帧、调色。
- 核心模块：工具轨、预览区、时间线、素材导入区、字幕区、导出工具栏、属性抽屉。
- 关键操作：导入素材、编辑多轨时间线、调整关键帧、调色、添加字幕、添加音乐、导出视频。
- 联动关系：联动素材工厂、AI内容工厂、定时发布。
- AI 能力：智能增强、字幕初稿、封面帧建议、节奏建议。
- 输出结果：视频工程、导出成片、项目快照。
- 验收要求：内容编辑流畅，导出结果稳定。

### 12.2 素材工厂

- 页面定位：统一素材资产库和批处理中心。
- 主要用户：内容创作者、团队协作成员。
- 局部导航：工作台、云端存储、处理历史、团队协作。
- 核心模块：素材网格、分类树、标签筛选、批量操作条、详情抽屉、共享区、历史区。
- 关键操作：上传文件、分类、打标签、搜索、预览、共享、批量下载、批量处理。
- 资产字段：类型、尺寸、时长、来源、创建人、版权状态、引用次数。
- 联动关系：联动视觉编辑器、下载器、局域网传输、AI内容工厂。
- AI 能力：自动标签、内容摘要、质量评分。
- 输出结果：素材资产、处理记录、共享记录。
- 验收要求：素材可检索、可复用、可追踪引用关系。

## 十三、功能需求，analytics 域

### 13.1 可视化实验室

- 页面定位：数据可视化分析构建器。
- 主要用户：数据分析师、运营经理。
- 局部导航：数据源、图表库、我的项目、文档。
- 时间范围：1H、1D、1W、1M。
- 核心模块：数据源面板、图表库、画布区、样式配置区、项目管理区。
- 关键操作：添加数据源、选择图表类型、配置样式、保存模板、发布结果。
- 联动关系：联动数据报告生成器和 Dashboard。
- AI 能力：图表建议、异常提示、可视化摘要。
- 输出结果：图表模板、项目快照、可共享分析视图。
- 验收要求：用户可在较短时间内完成图表构建。

### 13.2 竞争对手监控

- 页面定位：竞品内容与增长监控页。
- 主要用户：数据分析师、运营经理、内容策划。
- 页内标签：视频表现、增长趋势、内容分析。
- 核心模块：竞品列表、监控指标卡、趋势图、主题分析区、对比区。
- 关键操作：添加竞品、查看表现、查看趋势、多竞品对比、跳转蓝海分析。
- 联动关系：联动爆款标题、商品标题大师、蓝海分析。
- AI 能力：竞品摘要、主题归类、风险提醒。
- 输出结果：竞品监控列表、对比报告、趋势观察记录。
- 验收要求：竞品切换顺畅，对比信息清晰。

### 13.3 利润分析系统

- 页面定位：利润、成本、库存与 ROI 分析页。
- 主要用户：运营经理、数据分析师、选品人员。
- 局部导航：概览、利润分析、库存管理。
- 核心模块：利润表、费用参数区、库存列表、搜索栏、导出区。
- 关键操作：导入数据、计算净利润、计算 ROI、配置运费、汇率、佣金、平台费、导出报表。
- 联动关系：联动订单管理、报表中心、Dashboard。
- AI 能力：异常利润解释、波动原因摘要。
- 输出结果：利润报表、成本快照、库存风险提示。
- 验收要求：指标计算清晰且可导出。

### 13.4 蓝海分析

- 页面定位：市场机会发现页。
- 主要用户：数据分析师、运营经理、选品人员。
- 局部导航：机会矩阵、竞品监控、趋势发现。
- 核心模块：机会矩阵、趋势区、机会评分卡、报告入口。
- 关键操作：按品类筛选、按关键词筛选、按地区筛选、查看机会评分、生成调研报告。
- 联动关系：联动竞品监控、AI文案生成、爆款标题。
- AI 能力：AI 策略建议、调研报告生成、机会摘要。
- 输出结果：蓝海机会列表、机会矩阵、调研报告。
- 验收要求：机会评分可解释，结论可落到内容与运营动作。

### 13.5 数据报告生成器

## 十八、AI 集成需求

### 18.1 AI 集成目标

- 让 AI 成为系统底座。
- 让供应商切换不影响业务页。
- 让模型策略、角色策略和回退策略可运营化管理。

### 18.2 LiteLLM 传输层要求

- 使用 LiteLLM 作为统一 AI 传输层。
- 负责消息格式统一。
- 负责流式输出。
- 负责模型名映射。
- 负责错误归一化。
- 负责不同供应商调用体验统一。

### 18.3 ProviderAdapter 协议要求

ProviderAdapter 至少支持以下能力：

- 供应商标识读取
- 配置校验
- 连接测试
- 模型目录拉取
- 文本生成调用
- 流式调用
- 成本估算
- 错误归一化
- 速率限制处理
- 健康状态查询

### 18.4 首期支持的适配器类型

- OpenAI
- Anthropic
- Ollama
- OpenAI 兼容协议供应商

### 18.5 模型管理要求

- 支持全局默认模型。
- 支持页面级模型映射。
- 支持角色级模型映射。
- 支持温度、最大输出长度、超时、重试与流式开关配置。
- 支持标记推荐模型、低成本模型、本地模型、实验模型。

### 18.6 角色预设要求

- 所有 agent 角色必须以可编辑预设管理。
- 不允许只靠代码硬编码。
- 预设字段至少包括名称、描述、默认模型、系统提示、输出格式、风格要求、安全边界、适用页面、最大轮次、启用状态、版本号。

### 18.7 建议预置角色

- 爆款标题策划师
- 商品标题优化师
- 电商脚本编剧
- 文案合规审校员
- 蓝海机会顾问
- 竞品研究分析师
- 自动回复助手
- 客户跟进建议助手
- 报告总结助手

### 18.8 AI 输出治理要求

- 记录生成时间。
- 记录模型与供应商。
- 记录角色预设。
- 记录耗时与成本。
- 记录失败原因。
- 支持保存人工修订版。
- 支持原版与修订版对比。

### 18.9 AI 失败回退机制

- 单模型失败时自动重试。
- 多次失败后切换备用模型。
- 供应商失败后切换备用 ProviderAdapter。
- 回退链路必须可配置。
- 回退行为必须可追踪。

### 18.10 AI 合规与安全要求

- API Key 进入安全存储。
- 高风险输出进入规则检查。
- 页面级、角色级、团队级调用额度可配置。
- 敏感日志默认脱敏。

## 十九、技术架构概述

### 19.1 架构目标

- 支撑 26 个页面长期演进。
- 支撑多角色工作面。
- 支撑 AI 多供应商和多模型治理。
- 支撑主题切换、插件扩展和任务编排。

### 19.2 架构原则

- 使用 Clean Architecture 组织核心业务。
- 使用 MVVM 管理页面状态与视图逻辑。
- 使用 plugin shell 承载通用桌面结构。
- 使用 typed Qt signals 保证后台任务与 UI 的稳定通信。
- 使用依赖注入容器管理组件生命周期与依赖关系。
- 使用 Protocol 定义跨层接口契约，保证可替换性。
- 使用 Result 模式封装服务返回值，避免异常驱动控制流。

### 19.3 分层结构

- 表现层，负责 PySide6 页面与组件。
- 视图模型层，负责状态、命令、校验。
- 应用层，负责跨域编排。
- 领域层，负责业务规则与实体。
- 基础设施层，负责 SQLite、SQLAlchemy、网络、文件、日志、LiteLLM。

### 19.4 依赖注入与服务容器

当前 `TKOpsApp` 直接实例化所有组件并互相传递引用，随着系统增长会变成 God Object。本节规定依赖管理规范。

- 引入轻量 DI 容器或 ServiceLocator，统一管理组件的创建、引用和销毁。
- 所有核心组件（ConfigBus、ThemeEngine、Database、ServiceRegistry）通过容器注册和获取，不在 `app.py` 中直接持有公共属性。
- MainWindow 及页面依赖接口（Protocol）而非具体类。
- 容器负责启动校验：校验所有已注册服务是否正常初始化，校验失败时记录日志并降级运行。
- 容器负责关闭清理：应用退出时按依赖拓扑逆序释放资源。

### 19.5 plugin shell 要求

- 壳层必须包含 TitleBar。
- 壳层必须包含 CollapsibleSidebar。
- 壳层必须包含 PageHost。
- 壳层必须包含 DetailPanel。
- 壳层必须包含 StatusBar。
- 壳层必须支持最近访问、收藏和全局搜索。

### 19.6 ConfigBus 要求

- 统一广播配置变更。
- 支持字段级订阅。
- 支持持久化。
- 支持版本迁移。
- 主题、AI 配置、调度器和页面 ViewModel 可订阅。
- 读写操作必须线程安全，使用 QReadWriteLock 或 copy-on-write 策略。
- 订阅必须返回 Disposable 句柄，页面在 `on_deactivated` 时清理，防止订阅泄漏。
- 支持 `batch_update(callback)` 合并通知，避免一次性修改多个字段时触发多次回调。

### 19.7 ThemeEngine 要求

- 基于设计令牌输出主题。
- 负责浅色与深色切换。
- 负责注入专区角色色。
- 负责壳层与页面的统一风格。
- QSS 模板应拆分为模块化片段（base / controls / views / sidebar / pages），按需组装，避免单文件超长模板。

### 19.8 RouteRegistry 要求

- 所有正式页、hub 页、stub 页都必须注册。
- 路由字段至少包括标识、标题、图标、专区、权限、懒加载信息。
- 支持插件方式追加注册。
- 路由注册不得硬编码在 `app.py` 中。每个业务域目录下提供 `routes.py` 导出路由声明列表，`app.py` 通过自动发现机制扫描 `ui/pages/*/routes.py` 完成注册。
- 路由注册时必须校验：RouteId 唯一性、域组枚举值合法性、页面工厂模块路径有效性。

### 19.9 ViewModel 规范

所有页面必须通过 ViewModel 管理状态与业务逻辑，不允许页面 Widget 直接操作服务层或数据层。

- 定义 `BaseViewModel` 抽象基类，所有页面 ViewModel 继承此基类。
- `BaseViewModel` 规定以下接口：
  - `properties` — 可观察状态字典，变更时发出 Qt Signal。
  - `commands` — 可绑定命令字典，页面按钮直接关联。
  - `on_activated()` — 页面被激活时调用，负责加载数据和订阅配置。
  - `on_deactivated()` — 页面被切走时调用，负责清理订阅和释放资源。
  - `subscribe_config(key, handler)` — 快捷方法，订阅 ConfigBus 字段并自动纳入生命周期管理。
- 页面 Widget 只负责视图绑定和用户交互转发，不包含业务逻辑。

### 19.10 typed Qt signals 要求

- 所有长耗时任务必须后台执行。
- 所有任务需支持进度、成功、失败、日志信号。
- 复杂任务支持取消、暂停、恢复。

### 19.11 后台任务系统

本节补充任务系统的具体模型设计，不仅限于 Qt signals 通信。

- `TaskDescriptor` — 任务声明：类型、输入参数、优先级、重试策略、超时时间。
- `TaskExecutor` — 任务执行器：基于 QThreadPool 的线程池 + 信号桥接，限制最大并发数。
- `TaskStore` — 任务持久化：将任务状态写入 SQLite，应用重启后可恢复未完成队列。
- `TaskScheduler` — 任务调度：支持 cron 表达式、依赖拓扑排序、冲突检测。
- 任务间依赖通过 DAG 描述，调度器在执行前校验依赖是否满足。
- 资源限制：可配置最大并发任务数和单任务内存上限。

### 19.12 数据层要求

- 首期数据库为 SQLite。
- ORM 使用 SQLAlchemy。
- 文件资产与元数据分离存储。
- 关键对象支持未来迁移。

### 19.13 Repository 接口协议

所有数据访问必须通过 Repository 接口，业务层不直接操作 SQLAlchemy Session。

- 定义 `Repository[T]` Protocol，所有仓储实现此协议：
  - `get_by_id(id: str) -> T | None`
  - `list(filters: dict) -> list[T]`
  - `save(entity: T) -> None`
  - `delete(id: str) -> None`
- 每个核心实体对应一个 Repository 实现。
- Repository 实现内部封装 SQLAlchemy 会话管理、异常转换和事务边界。
- 服务层通过 Protocol 引用 Repository，方便替换为内存实现用于开发调试。

### 19.14 页面复杂度分级

26 个页面复杂度差异巨大，用同一套框架模式会导致简单页面写过多 boilerplate、复杂页面框架约束不够。按复杂度分级提供页面基类：

- `SimplePage` — 适用于表格 + 筛选 + CRUD 场景，如网络诊断、下载器、局域网传输、任务队列。自动绑定表格数据、筛选栏和标准操作条。
- `WorkspacePage` — 适用于多面板 + 工具栏 + 属性抽屉场景，如账号管理、CRM、素材工厂、AI供应商配置。内置 DetailPanel 集成和多面板布局。
- `CanvasPage` — 适用于画布 + 节点/时间轴 + 自定义渲染场景，如视觉编辑器、AI内容工厂、可视化实验室。提供画布缩放、节点拖拽、时间轴控件的基础支持。

所有页面基类均继承自通用 `BasePage`，共享路由注册、主题刷新、ViewModel 绑定和生命周期钩子。

### 19.15 错误边界与降级策略

- 每个页面的 `on_activated` 必须在错误边界内运行，单个页面崩溃不得拖垮整个壳层。页面加载失败时展示错误占位页，提供重试和返回入口。
- 服务层方法返回 `Result[T, Error]` 类型而非直接抛异常，让调用方显式处理失败路径。
- AI 供应商全部不可用时，依赖 AI 的页面展示降级提示而非空白或报错。
- 引入 `SessionRestore` 机制：页面在 `on_deactivated` 时序列化关键表单状态到本地存储，应用重启后可恢复到上次页面和未完成的表单内容。

### 19.16 插件扩展要求

- 页面可通过插件接入。
- AI 供应商可通过插件接入。
- 报告模板与角色预设可通过插件接入。
- 插件需支持加载、卸载、校验与健康检查。

### 19.17 日志规范

- 日志格式统一为结构化 JSON。
- 日志分为三类：business（业务操作）、system（系统运行）、audit（审计记录）。
- 敏感信息默认脱敏。
- AI 调用日志记录模型、耗时、成本、角色预设和失败原因。

### 19.18 密钥安全实现

- API Key 和敏感凭据统一通过 Windows Credential Manager（`win32cred`）存储。
- 若 Windows Credential Manager 不可用，降级为加密 SQLite 列（AES-256）。
- 密钥在 UI 层始终脱敏展示，不可明文回显。
- 密钥变更纳入审计日志。

### 19.19 国际化预留

- UI 字符串统一通过 `QCoreApplication.translate()` 或 `tr()` 包裹。
- 首期仅支持中文，但代码结构上为多语言扩展预留入口。
- 不在业务代码中硬编码用户可见文本。

## 二十一、用户流程

### 21.1 首次启动流程

1. 用户打开应用。
2. 系统检查初始化标记。
3. 若未初始化，则进入 Setup Wizard。
4. 用户配置 AI Provider。
5. 用户配置 TTS 服务。
6. 用户完成浏览器与网络基础设置。
7. 系统保存配置并完成校验。
8. 用户进入 TK-OPS Dashboard。

### 21.2 每日运营流程

1. 用户打开 Dashboard。
2. 查看账号、任务、AI、客户和异常摘要。
3. 进入目标专区。
4. 在账号管理确认账号状态。
5. 在分析区完成竞品、蓝海、利润、互动判断。
6. 在 AI 区和内容区完成生成、提取、加工。
7. 在自动化区创建采集、回复和发布任务。
8. 在 CRM 中跟进高价值客户。
9. 在数据报告生成器中完成汇报输出。

### 21.3 AI 内容生产流程

1. 从竞争对手监控和蓝海分析获取机会。
2. 用爆款标题和商品标题大师生成标题。
3. 用AI文案生成产出内容草稿。
4. 用脚本提取工具参考已有视频结构。
5. 用AI内容工厂编排流程。
6. 用视觉编辑器制作成片。
7. 用素材工厂归档资产。
8. 用定时发布安排上线。

### 21.4 自动化运营流程

1. 系统管理员配置 AI 供应商与回退链路。
2. 自动化运营专员在自动回复控制台配置规则。
3. 在数据采集助手创建采集任务。
4. 在任务调度中安排计划。
5. 在任务队列中追踪执行状态。
6. 在 Dashboard 中查看摘要和异常。
7. 在 CRM 中消费相关客户结果。

### 21.5 数据复盘流程

1. 在可视化实验室构建图表。
2. 在竞争对手监控、蓝海分析、利润分析系统、互动分析中补充证据。
3. 在数据报告生成器中生成正式报告。
4. 使用 AI 生成摘要和结论。
5. 导出并发送报告。
6. 将关键结论回流到 Dashboard 和运营动作。

## 二十二、非功能需求

### 22.1 性能

- 首次启动在合理时间内完成。

- 失败任务必须有原因。
- 关键任务必须可重试。
- 应用异常退出后尽量保留关键上下文。

- API Key 进入安全存储。
- 账号与客户敏感信息默认脱敏。
- 页面、按钮、导出、配置修改都要纳入权限控制。

- 新供应商可按 ProviderAdapter 接入。

### 22.5 可维护性

- 配置统一走 ConfigBus。

### 22.6 可观测性

- 有任务日志。
- 有网络诊断日志。
- 有 AI 调用日志。

### 22.7 可用性

### 22.8 可伸缩性

- 支持更多业务域扩展。
- 支持更多模型接入。
- 支持更多插件接入。

## 二十六、风险与应对

### 26.1 AI 供应商不稳定风险

- 通过 LiteLLM 与 ProviderAdapter 做多供应商切换。
- 通过回退链路支持备用模型。

### 26.2 页面范围过大风险

- 通过 plugin shell、RouteRegistry 和里程碑拆分管理范围。
- 对尚未落地能力先用 hub 页或 stub 页占位。

### 26.3 多角色复杂度过高风险

- 通过五大专区和全局搜索降低理解成本。
- 通过 Dashboard 统一总入口。

### 26.4 自动化误操作风险

- 引入人工审校、限频、日志、回滚和人工接管。

### 26.5 数据与密钥安全风险

- 敏感信息脱敏。
- 密钥进入安全存储。
- 权限细分到页面和操作层。

## 二十七、结论

TK-OPS 的目标不是简单复刻若干页面。
它要成为一个围绕 TikTok Shop 运营场景构建的企业级桌面产品。
这份 PRD 明确了产品目标、目标用户、愿景、二十六个页面、九大域组、五大专区、AI 架构、技术架构、数据模型、用户流程、优先级和里程碑。
后续设计、开发和验收都应以本文件为主依据。
