# TK-OPS 全量后端接线与交互补全设计规格

## 1. 文档信息

- 日期：2026-03-22
- 主题：全量接线、交互清查、分层补后端、逐页从前到后打通
- 范围：将真实运行链路中的页面从写死数据/占位数据迁移到真实后端与数据库驱动，并保证所有可点击交互有完整逻辑链
- 语言与协作方式：中文、逐页推进、自查自检

## 2. 背景

当前仓库已经完成了一轮前端样式恢复与多轮 UI 可用性修复，但仍存在两个更深层的问题：

1. 大量页面仍然依赖前端模板中的写死数据、占位卡片、假统计和静态洞察。
2. 不少按钮、筛选、切换、详情联动、弹层、下拉、分析操作在视觉上可点击，但逻辑链不完整，或只提供假反馈。

用户已经明确要求：

1. 全量接线 + 交互清查 + 分层补后端。
2. 逐页从前到后打通，而不是只挑几个页面补丁式修复。
3. 程序要完整稳定运行，逻辑链必须打通，尽量不出 bug。
4. 所有能点的按钮、能交互的地方都要跑通，不能有遗漏。
5. 如业务需要，可以新增表、修改表结构，并允许创建开发测试数据验证真实接入。

## 3. 核心目标

本轮目标不是单纯改样式，而是把当前桌面应用提升到“前端展示、交互行为、后端接口、数据库数据”一致的状态。

具体目标：

1. 所有主页面的主数据来源必须是真实数据库或真实后端聚合，而不是前端硬编码数组。
2. 所有可点击交互必须能明确落到：查询、变更、任务发起、导出、详情更新或受控降级中的一种。
3. 页面之间的数据刷新、侧栏详情、状态栏、通知中心、任务队列、分析页联动必须使用真实数据回流，而不是假刷新。
4. 页面缺少真实业务模型时，必须补模型 / 服务 / bridge，或明确将页面语义收口为当前数据库真实可支撑的内容，禁止伪造业务数据。
5. 每一页都要有自查自检与回归标准，保证逐页打通后不会留下“表面能点、实际没逻辑”的漏项。

## 4. 非目标

以下内容不在本轮范围内：

1. 不引入前端框架、构建工具或重写为 SPA 框架。
2. 不为了“图表好看”而引入新的大型图表依赖；优先基于现有静态资源、canvas、CSS 和数据结构改造。
3. 不为不存在的真实业务数据伪造销售额、ROI、订单额等指标。
4. 不优先处理未接线旧目录，如 `desktop_app/assets/routes/` 和 `desktop_app/assets/app_shell.js`。

## 5. 当前系统能力盘点

### 5.1 已有数据库模型

当前 `desktop_app/database/models.py` 中已有的持久化实体：

1. `Account`
2. `Group`
3. `Device`
4. `Task`
5. `AIProvider`
6. `Asset`
7. `AppSetting`

这些模型足以支撑当前系统的基础 CRUD、设置、任务队列、素材、AI 供应商等主链路页面。

### 5.2 已有后端服务与桥接能力

`desktop_app/ui/bridge.py` 已经暴露并较成熟支持的能力包括：

1. 账号管理
2. 分组管理
3. 设备管理
4. 任务管理
5. AI 供应商管理
6. 素材管理
7. 设置与主题类能力
8. 许可证、更新、AI 聊天、使用统计等系统能力

这意味着第一批页面可以直接走现有 bridge + service + repository 真接线。

### 5.3 当前能力缺口

分析页、实验页、工作流页、报表页、创作页中有大量内容仍是工厂模板或前端推导值，后端缺口主要体现在：

1. 缺少稳定的聚合服务层。
2. 缺少对应的分析快照、实验项目、工作流定义、运行记录等持久化实体。
3. 缺少统一的页面交互矩阵与“每个按钮落到哪里”的映射规范。

## 6. 总体策略

采用“全量接线 + 交互清查 + 分层补后端 + 逐页从前到后打通”的组合策略。

### 6.1 分层策略

#### 第一层：已有 CRUD 主链路页面

优先覆盖：

1. 概览数据看板
2. 账号管理
3. 分组管理
4. 设备管理
5. AI 供应商配置
6. 任务队列
7. 素材中心
8. 系统设置 / 日志中心 / 版本升级 / 网络诊断等系统页

这些页面应优先做到“真数据 + 真交互 + 真反馈”。

#### 第二层：分析与看板页面

覆盖：

1. 可视化实验室
2. 竞品对手监控
3. 流量看板
4. 利润分析
5. 蓝海分析
6. 报表中心
7. 互动分析
8. 电商转化
9. 粉丝画像

这些页面的核心工作是：

1. 清除写死指标与假结论。
2. 用现有实体做真实聚合，必要时新增聚合接口。
3. 当现有数据库无法支撑某些指标时，重新定义页面语义或补新表，禁止继续造数。

#### 第三层：创作 / 工作流 / 实验页面

覆盖：

1. 视频剪辑
2. 视频编辑器
3. AI 生成相关页面
4. 工作流相关页面
5. 工具台 / 控制台 / 实验操作页

这些页面的重点不是外观，而是保存、加载、历史、切换、运行、导出、回放等行为的真实性。

### 6.2 逐页推进顺序

执行顺序固定为：

1. 先做公共交互底座。
2. 再按导航与主链路逐页打通第一层页面。
3. 再逐页打通分析区页面。
4. 最后打通创作 / 工作流 / 实验页面。

## 7. 公共交互底座设计

在逐页推进前，必须先统一公共行为，否则后续每页都可能重复出现半通不通的问题。

### 7.1 需要统一的公共行为

1. 按钮点击反馈与禁用态。
2. 下拉、筛选、分页、切换、tab、segmented 控件。
3. 详情侧栏与 detailHost 更新机制。
4. 通知中心、状态面板、浮层、弹层。
5. 加载态、空状态、错误态、成功态。
6. 数据变更后 UI 回流刷新。

### 7.2 公共底座的目标

1. 所有可见交互点都能被统一登记并绑定。
2. 所有失败路径都能明确反馈，而不是静默无效。
3. 页面在刷新、重复进入、窗口变化后不丢交互状态，不重复绑定。

## 8. 页面分层打通设计

### 8.1 第一层：已有 CRUD 页面

#### 8.1.1 账号 / 分组 / 设备 / 任务 / AI 供应商 / 素材 / 设置

每个页面都应具备：

1. 真列表：来自真实 `list*` bridge slot。
2. 真详情：来自真实对象或当前选中对象回流。
3. 真表单：新建、编辑、删除、启用、绑定都落真实 mutation。
4. 真筛选：前端本地视图筛选与后端数据源一致。
5. 真分页：若暂未做服务端分页，也至少基于真实数据做前端分页，而不是假页脚。

#### 8.1.2 概览数据看板

概览页必须不再直接展示模板中的静态 KPI，而应由数据库真实计数与状态聚合构成，包括：

1. 账号状态统计。
2. 任务状态统计。
3. 设备状态统计。
4. 供应商、素材、设置、通知、最近操作等系统真实摘要。

### 8.2 第二层：分析与看板页面

#### 8.2.1 真实聚合原则

分析页所有指标都必须先回答：

1. 数据源是什么？
2. 聚合口径是什么？
3. 这个页面的点击动作是在查询、切换维度，还是发起后台任务？

#### 8.2.2 禁止伪造业务指标

例如利润、ROI、订单额、真实转化率，如果当前数据库并没有订单、交易、履约、投流等模型，则不允许继续展示看似真实的业务数值。

允许的解决方式只有两种：

1. 新增相应模型、迁移、服务与页面接线，形成真实数据闭环。
2. 将页面语义临时收口为“运营过程指标”或“任务效率指标”，只展示现有数据库真实可以支撑的数据。

#### 8.2.3 分析页交互标准

电商转化、粉丝画像、流量来源、报表步骤、利润结构等交互必须：

1. 单实例反馈，不重复弹框。
2. 点击后更新真实侧栏 / 详情容器 / 查询结果。
3. 若动作本质是后台任务，则落任务队列而不是假弹窗。

### 8.3 第三层：创作 / 工作流 / 实验页面

这类页面最容易只做原型感，因此本轮要求更严格：

1. 保存操作必须落数据库。
2. 列表、草稿、模板、运行记录必须真实可回读。
3. 运行类动作必须要么真实执行业务，要么发起真实任务并追踪状态。
4. 不能再保留“点击后 toast 一下但无任何记录”的伪交互。

## 9. 按钮与交互行为映射规范

页面上的每一个交互点都必须落到以下四类之一。

### 9.1 直接写现有表

适用于：

1. 新建 / 编辑 / 删除账号
2. 新建 / 编辑 / 删除分组
3. 新建 / 编辑 / 删除设备
4. 新建 / 编辑 / 删除任务
5. 新建 / 编辑 / 删除 AI 供应商
6. 素材增删改查
7. 设置项保存

### 9.2 落任务队列

适用于复杂或异步动作，例如：

1. 测试连接
2. 批量检测环境
3. 导出分析结果
4. 生成报表
5. 发起实验验证
6. 同步到其他页面或工作台
7. 隔离环境检查、复核、回放等后台动作

这些交互应统一映射到真实 `Task`，由任务状态回流 UI。

### 9.3 走聚合查询接口

适用于：

1. 趋势切换
2. 维度切换
3. 画像切换
4. 转化阶段明细
5. 利润结构拆解
6. 流量来源比较

这类动作本质是查询，不应写表或伪造值，而应由聚合 service 返回真实结果。

### 9.4 受控降级

仅当某功能当前业务上确实不存在完整实现，且短期不做新增业务时允许使用。

要求：

1. 必须明确禁用，不允许看起来能点。
2. 必须给出说明，告诉用户为什么当前不可用。
3. 不允许继续保留“点了没反应”的状态。

用户当前要求严格全量跑通，因此受控降级只允许用于真正超出现有业务定义、且已与设计文档明确记录的极少数入口。

## 10. 新增表 / 改表结构策略

### 10.1 可以直接继续使用的现有表

1. `accounts`
2. `groups`
3. `devices`
4. `tasks`
5. `ai_providers`
6. `assets`
7. `app_settings`

### 10.2 建议新增的持久化能力

根据页面能力缺口，建议新增以下类型的实体（命名可调整，但职责需保留）：

1. `analysis_snapshots`
   - 保存分析页生成的真实统计快照、维度结果、筛选状态、摘要结论
2. `report_runs`
   - 保存报表生成请求、运行状态、输出结果、关联任务
3. `workflow_definitions`
   - 保存工作流结构、步骤定义、输入配置
4. `workflow_runs`
   - 保存工作流执行记录、状态、日志摘要、输出引用
5. `experiment_projects`
   - 保存可视化实验项目与其业务目标
6. `experiment_views`
   - 保存实验视图配置、图表组合、标签、交互设定
7. `activity_logs` 或等价事件表
   - 为通知中心、最近动作、页面右侧摘要、状态面板提供真实来源

### 10.3 改表结构的原则

允许在现有表中补字段，但必须遵循：

1. 只为当前真实页面能力服务。
2. 必须配套 Alembic migration。
3. 必须同步更新 ORM、service、bridge 和测试。
4. 新字段不能只为前端显示存在而没有实际语义。

## 11. 开发测试数据策略

用户已授权为了验证真实接入而创建测试数据，但要求这些数据也必须通过真实链路存在。

### 11.1 允许的方式

1. 数据库 seed 脚本
2. 启动时开发环境填充
3. 测试夹具或测试专用初始化函数
4. 开发模式下的示例记录创建工具

### 11.2 不允许的方式

1. 前端 factories 内部硬编码数组作为“伪真数据”
2. loader 里直接写死统计值冒充后端返回
3. 点击行为只 toast 成功但不写入数据库或任务队列

## 12. 逐页自查自检机制

每个页面在完成接线前后都必须建立一张交互矩阵，至少包含：

1. 页面主数据来源
2. 页面 secondary 数据来源
3. 主按钮列表
4. 次按钮列表
5. 卡片 / 行 / tab / 筛选 / 下拉 / 分页 / 详情切换列表
6. 每个交互调用的接口或任务发起路径
7. 成功反馈
8. 失败反馈
9. 空状态反馈
10. 刷新 / 重进 / 重复点击行为

页面完成标准：

1. 所有可见数据均可追溯到真实来源。
2. 所有可见交互均可追溯到真实后端或真实任务。
3. 不存在看起来可点击但无效果的控件。
4. 不存在只在首次进入能用、刷新后失效的交互。

## 13. 测试与验证策略

### 13.1 自动化验证

需要逐步建立三类测试：

1. 前端结构/选择器基线测试
2. 后端 bridge/service/repository 行为测试
3. 针对新增聚合服务和新增模型的回归测试

### 13.2 手动验证

每个页面至少验证：

1. 打开页面
2. 主数据加载
3. 主按钮
4. 次按钮
5. 行点击 / 卡片点击
6. 筛选 / 下拉 / 分页
7. 新增 / 编辑 / 删除
8. 空状态
9. 错误状态
10. 页面刷新后再次进入

### 13.3 稳定性要求

用户明确要求完整运行与逻辑链打通，因此在每个阶段结束时必须执行：

1. 页面级验证
2. 关键路径验证
3. 全量测试
4. 桌面应用实际运行验证

## 15. 成功标准

若本轮实施完成，系统应满足：

1. 页面主数据不再依赖写死数组或占位数据。
2. 所有可点击入口都有真实逻辑链。
3. 所有关键动作都能落到真实数据库、真实任务队列或真实聚合接口。
4. 不存在“看起来能点但无效果”的按钮、筛选或弹层。
5. 页面切换、刷新、重进、重复点击后状态稳定。
6. 分析页不再伪造业务数据，无法真实支撑的内容要么补模型，要么明确收口。
7. 每一页都有可追溯的数据来源、交互映射与验证结果。

本规格即作为下一阶段 implementation plan 的依据。
