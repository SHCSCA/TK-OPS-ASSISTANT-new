# 个人版全量迁移与 EXE 交付设计（2026-04-02）

> 基于已审批计划：`docs/superpowers/plans/2026-04-02-个人版全量迁移与EXE交付计划.md`

## 1. 设计目标

## 1.1 业务目标

1. 新架构下完整承接旧系统核心能力，确保个人用户可持续使用。
2. 交付可安装、可启动、可执行关键链路的 Windows EXE。
3. 从“可演示”提升到“可发布”：真实数据、可恢复异常、可追踪日志。

## 1.2 工程目标

1. 降低 `desktop_app/` 对主链路的隐性依赖。
2. 强化 `apps/desktop` 与 `apps/py-runtime` 契约边界（API/事件/错误 envelope）。
3. 构建可回归、可回滚、可持续迭代的发布流程。

---

## 2. 架构与边界设计

## 2.1 目标架构（运行时）

```text
[Tauri Desktop Host]
        |
        | HTTP / WS
        v
[Python Runtime (FastAPI)] ----> [SQLite + SQLAlchemy + Alembic]
        |
        +----> [Domain Services / Repositories]
```

- `apps/desktop`：页面渲染、交互状态、运行时健康探测、用户反馈。
- `apps/py-runtime`：业务 API、任务编排、数据持久化、运行时状态。
- `desktop_app/`：仅作为迁移期代码来源，禁止新增壳层能力。

## 2.2 关键边界约束

1. 前端业务调用仅通过统一数据访问层，不允许散落直连。
2. 后端返回统一 envelope：
   - 成功：`{ ok: true, data }`
   - 失败：`{ ok: false, error, code?, trace_id? }`
3. 所有写操作完成后必须触发可观测变更事件（用于缓存失效和 UI 刷新）。

---

## 3. 迁移域模型（Domain Slice）

## 3.1 迁移优先级

- **P0（阻塞发布）**：Dashboard、账号管理、Provider、任务队列、任务调度、AI 文案、Setup、设置、视频编辑主链路。
- **P1（高价值增强）**：账号导入向导、完整审计页、环境绑定深链路。
- **P2（发布后迭代）**：低频功能与非关键体验优化。

## 3.2 迁移完成定义（Definition of Migrated）

某能力仅在以下 6 项全部满足时，才判定“完成迁移”：

1. 新架构页面可进入并可完成主操作。
2. 读写均来自 runtime 真实数据，不使用伪业务数据。
3. 异常场景有用户可理解反馈与恢复建议。
4. 对应 API/服务层有契约测试或行为测试。
5. 打包后 smoke 中可复现该能力（非仅 dev 模式）。
6. 文档已更新（变更说明 + 已知限制）。

---

## 4. 前端设计（apps/desktop）

## 4.1 模块拆分原则

1. 路由、页面装载、交互绑定、数据访问分层，避免巨石文件继续膨胀。
2. 所有业务错误必须有 UI 可见反馈（toast/banner/inline）。
3. 主题与样式统一使用设计令牌，不新增硬编码色值。

## 4.2 运行时状态机（前端视角）

```text
BOOTING -> HEALTHY -> DEGRADED -> RECOVERING -> HEALTHY
             |             |
             +----> ERROR -+
```

- `BOOTING`：壳启动，等待 runtime 可用。
- `DEGRADED`：部分 API 失败，但保留最小可用操作。
- `RECOVERING`：自动/手动重试中。
- `ERROR`：不可恢复故障，给出修复建议与日志路径。

## 4.3 页面迁移模板

每个页面按统一模板迁移：

1. route 元数据就位。
2. loader 接入真实 API。
3. bindings 绑定创建/编辑/删除/筛选/详情。
4. 主操作、异常态、空态齐全。
5. 添加页面级回归测试。

---

## 5. 后端设计（apps/py-runtime）

## 5.1 API 设计准则

1. 路由层只做协议转换，不承载业务规则。
2. 业务规则落在 service 层，数据读写落在 repository 层。
3. 错误分类（参数错误/业务错误/系统错误）统一映射到 envelope。

## 5.2 数据与迁移

1. Alembic 迁移必须支持增量升级，不依赖“全新空库”假设。
2. 对历史库结构差异提供兼容分支与修复脚本。
3. 关键表变更需附带回填逻辑与回退策略。

## 5.3 Sidecar 稳定性设计

1. runtime 启动超时与健康检查机制。
2. 异常退出探测、受控重启与次数阈值。
3. 失败后向前端输出可操作信息（重试、日志、配置修复提示）。

---

## 6. EXE 交付设计

## 6.1 发布链路

1. 版本号与变更说明对齐。
2. 生成桌面产物并注入 runtime 依赖。
3. 产出安装包并执行安装后回归。
4. 输出发布记录（已知问题、回滚方式、验证结果）。

## 6.2 安装后验收清单（个人用户）

1. 首次启动成功（含初始化向导）。
2. 关键页面可打开并完成 1 条业务主操作。
3. 任务/导出等异步能力具备可见状态反馈。
4. 配置持久化有效（重启后保留）。
5. 错误发生时可见并可恢复。

---

## 7. 测试与质量门禁

## 7.1 测试分层

1. **契约测试**：API 返回结构、字段兼容、错误 envelope。
2. **服务测试**：service/repository 行为真实性与回滚能力。
3. **页面交互测试**：主动作、批量动作、异常反馈、空态。
4. **启动/发布 smoke**：sidecar 生命周期、打包后关键链路。

## 7.2 最低发布门禁

- 变更相关自动化测试全部通过。
- 启动与主链路 smoke 通过。
- 至少 1 轮安装后回归通过并有记录。
- 发布记录文档完成。

---

## 8. 风险与缓解

1. **旧依赖剥离不彻底**
   - 缓解：建立依赖地图，按域断链，禁止新增反向依赖。
2. **运行时异常定位困难**
   - 缓解：trace_id + 结构化日志 + 前端可见错误摘要。
3. **打包后行为偏差**
   - 缓解：将安装后回归纳入硬门禁，不通过不得发布。
4. **范围持续膨胀**
   - 缓解：维持 P0/P1/P2 冻结，新增需求滚入下一迭代。

## 9. 实施阶段与里程碑

## M1：能力盘点冻结（1 周）

- 输出全量盘点与个人版白名单。
- 验收：无“未分类能力”，优先级明确。

## M2：P0 全量迁移闭环（2~3 周）

- 完成阻塞发布能力的迁移与测试。
- 验收：P0 主链路全部可执行。

## M3：稳定性与发布治理（1~2 周）

- 完成 sidecar 稳定性强化与安装后回归体系。
- 验收：可连续构建、可安装、可回归。

---

## 10. 实施前置清单（进入编码前必须确认）

1. 页面迁移矩阵已更新到最新真实状态。
2. API 契约差异清单已冻结。
3. 数据迁移风险项（历史库/兼容）已标注。
4. 发布脚本与版本策略已对齐。

---

## 11. 结论

本设计将“全量迁移 + 个人版收敛 + EXE 交付”拆为可执行、可验证、可回退的工程路径。
在不引入团队协作复杂度的前提下，以 P0 主链路和发布门禁为核心，优先确保可用、稳定、可交付。

## 12. 基于 4 月 1 日文档的深化设计补充

本节将 4 月 1 日文档中的关键约束提升为当前 spec 的实施细则，防止迁移过程偏航。

## 12.1 单壳切换实施细则

1. 运行、联调、打包、发布四条链路统一以 `apps/desktop + apps/py-runtime` 为唯一主路径。
2. 旧 `desktop_app` 禁止新增页面、桥接 slot、临时功能，避免历史包袱继续增长。
3. 发布前检查项新增“单壳校验”：产物内不得包含旧壳可执行入口。

## 12.2 小文件与职责边界实施细则

1. 前端：路由、API 客户端、页面 ViewModel、UI 组件分层，避免回到单文件大杂烩。
2. 后端：按路由层/服务层/仓储层分离，限制单文件持续膨胀。
3. 对超阈值文件执行“先拆类型 -> 再拆客户端/服务 -> 再拆处理器/组件”固定顺序。

## 12.3 协议边界深化（HTTP / WS / 健康探测）

1. `/health` 语义：不仅用于探活，还需返回 runtime 基础边界元信息（版本、协议、状态）。
2. WS 语义：连接建立后先握手再进入业务事件流，避免频道间协议分叉。
3. token 语义：`session_token` 为统一优先凭据，HTTP 与 WS 鉴权口径保持一致。

## 12.4 迁移执行工作包（WP）拆分

### WP-1：协议与入口收口
- 目标：统一入口、统一协议、统一探活口径。
- 产物：协议文档更新、入口文档更新、健康探测与握手契约测试。

### WP-2：P0 业务域闭环
- 目标：完成 P0 页面“真实数据 + 可恢复交互 + 测试护栏”。
- 产物：页面迁移 PR、契约测试、页面交互回归测试。

### WP-3：sidecar 稳定性
- 目标：提升 runtime 启停可靠性与异常恢复能力。
- 产物：启动/重启/异常恢复 smoke 清单与脚本化验证记录。

### WP-4：发布与回滚治理
- 目标：将 EXE 交付固化为可重复流程。
- 产物：发布说明模板、安装后回归记录、回滚演练记录。

## 12.5 量化验收指标（新增）

1. P0 页面迁移完成率：100%（以“迁移完成定义”为准，不按页面可见性计数）。
2. 阻塞发布缺陷：0 个。
3. 安装后关键主链路回归：100% 通过。
4. 协议文档与实现偏差项：0 个（每次发布前核对）。
