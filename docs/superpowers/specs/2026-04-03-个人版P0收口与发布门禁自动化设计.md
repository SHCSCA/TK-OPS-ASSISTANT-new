# 个人版 P0 收口与发布门禁自动化设计（2026-04-03）

## 1. 设计目标

### 1.1 业务目标

- 个人版用户在“安装 -> 首启 -> 配置 -> 关键操作 -> 问题恢复”链路中获得稳定、可恢复、可追踪体验。
- 账号管理从“可用”提升为“可运营”：导入、审计、生命周期规则完整闭环。
- 发布流程可重复执行，且质量门禁不依赖人工记忆。

### 1.2 工程目标

- 统一发布前检查入口，形成可脚本化的通过/失败判定。
- 继续执行单壳与协议边界约束：`apps/desktop + apps/py-runtime` 为主链路，`desktop_app` 仅作迁移来源。
- 所有高风险改动必须有对应测试与文档同步。

## 2. 架构与边界

### 2.1 总体架构

- 前端：`apps/desktop` 承担用户交互与运行时状态可视化。
- 运行时：`apps/py-runtime` 提供 HTTP/WS 契约能力与业务服务。
- 发布链路：`scripts/*` 负责构建、smoke、打包、验收门禁。
- 文档层：`docs/releases/*` 与 `README.md` 提供可执行操作说明。

### 2.2 边界约束

- 不新增对旧 PySide 壳的运行时依赖。
- HTTP/WS 鉴权规则保持统一，不允许接口语义漂移。
- 错误信息保持中文可理解，不暴露 traceback 给最终用户。

## 3. 关键设计

### 3.1 发布前门禁脚本（`preflight-gate.ps1`）

- 输入模式：
  - `-Quick`：开发阶段快速检查（关键 pytest + smoke）。
  - `-Full`：发布前完整检查（pytest、typecheck、build、smoke、release 链路检查）。
- 输出：
  - 每个检查项独立输出开始/结束/耗时/结果。
  - 末尾输出汇总表和总退出码（任一失败返回非 0）。
- 原则：
  - 不替代已有构建脚本，而是编排并收敛入口。
  - 支持在 CI/本地一致调用。

### 3.2 账号域二三阶段收口

- 二阶段（导入向导）：
  - 前端维护导入流程状态机：文件选择 -> 字段映射 -> 预检查 -> 执行结果。
  - 后端提供预检查与冲突摘要接口，返回统一 envelope。
- 三阶段（审计与生命周期）：
  - 审计查询能力从“摘要”扩展到“可筛选历史”。
  - 生命周期规则统一动作边界：停用、归档、恢复、删除，避免语义重叠。

### 3.3 稳定性与恢复

- sidecar 管理增强：启动超时、异常退出重拉起、状态可视反馈。
- 关键失败路径要求：
  - UI 可见错误
  - 提供可执行恢复动作（重试/修复建议/日志定位提示）

## 4. 数据流与错误处理

### 4.1 数据流

- UI 仅通过统一 API 模块访问后端（避免散落调用）。
- Runtime 返回 envelope：`{ ok: true, data } | { ok: false, error }`。
- 数据变更需触发刷新链路，保证前端缓存失效后重取。

### 4.2 错误处理

- 业务错误转中文提示，技术细节写入日志。
- 门禁脚本失败时提供失败阶段与建议动作，不吞错误。

## 5. 测试与验收策略

### 5.1 测试分层

- 协议与运行时：`tests/test_runtime_api.py`、`tests/test_runtime_ws_auth_contract.py`
- 账号垂直切片：`tests/test_accounts_vertical_slice.py`
- 启动与发布链路：`tests/test_startup_*`、`tests/test_release_chain_single_shell.py`

### 5.2 验收门禁

- `scripts/preflight-gate.ps1 -Quick`：日常开发门禁。
- `scripts/preflight-gate.ps1 -Full`：发布前必过门禁。
- 安装后主链路 smoke 通过，且发布记录完整。

## 6. 风险与缓解

1. 风险：门禁脚本过重影响迭代速度。  
   缓解：双档位（Quick/Full）+ 分阶段运行。
2. 风险：账号域扩展引入接口不兼容。  
   缓解：先补契约测试，再改实现。
3. 风险：发布环境差异导致本地通过但安装后失败。  
   缓解：把安装后 smoke 纳入 Full 门禁与 release 记录。

## 7. 实施顺序

1. 先落地发布前门禁自动化（快速提质、低风险）。  
2. 再推进账号域二三阶段收口（业务价值最高）。  
3. 最后收口稳定性与发布治理，并完成完整验收。  

## 8. 完成判定

- 方向 A/B/C/D 均达到目标完成度。
- 有可复现命令与可追踪日志证明质量门禁通过。
- 文档、脚本、测试、运行行为保持一致。
