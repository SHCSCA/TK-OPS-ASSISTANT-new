## 1. 文档信息

- 日期：2026-03-22
- 主题：analyst 页面与通知中心后端全驱动化，以及配套测试数据补齐
- 范围：在已完成主链路接线基础上，继续清理仍带 prototype 痕迹的 analyst / analytics / notification 展示面，要求页面显示内容只来自真实数据库记录、真实聚合结果或真实空状态
- 协作方式：中文，直接实施，优先保证真实性与测试稳定性

## 2. 背景

前一轮已经完成：

1. CRUD 主页面接线
2. task-backed actions 闭环
3. analytics / reports / workflows / experiments 的最小持久化能力
4. dev seed 的最小可用版本
5. 全量 `tests/` 通过

但当前仍有一类问题没有彻底消失：

1. 通知中心仍然依赖前端定时注入模拟消息
2. analyst / analytics 类页面仍保留较多静态示例卡片、示例指标、示例建议
3. 某些默认空态和变量命名仍带有 placeholder / prototype 语义
4. 当前 seed 数据足以支撑测试，但还不足以把所有 analyst 页面首屏填成“真实可看的页面”

用户进一步明确要求：

1. 全部改为后端全驱动
2. 可以在数据库中写入测试数据，供界面查看和自动化测试使用
3. 不接受继续保留前端伪造通知、伪造洞察、伪造业务值

## 3. 目标

本轮目标是把“显示层最后一批假感来源”彻底收口。

完成后应满足：

1. 通知中心只展示真实记录映射出的通知项
2. analyst / analytics 页面主指标、主列表、主卡片尽量由真实聚合或真实持久化记录回填
3. 无真实来源的数据区域必须显示明确空状态，而不是示例业务值
4. 开发测试环境可通过 seed 写入一套足够支撑页面查看的真实记录
5. 自动化测试能约束这些页面不再退回到模拟通知和静态示例数据

## 4. 非目标

本轮不做：

1. 新增不存在的真实业务域模型，如订单、GMV、投流消耗、利润核算流水
2. 重写页面布局或更换现有 analyst 工厂体系
3. 为了填满界面而伪造业务结论

## 5. 总体方案

采用“后端真实来源优先，前端工厂降级为骨架模板，seed 补齐查看样本”的方案。

### 5.1 通知中心

通知中心改为读取真实来源：

1. `activity_logs` 作为主要通知来源
2. `tasks` 作为补充来源，用于映射任务完成、失败、排队等事件
3. 如有必要，可由 bridge 聚合成统一通知 payload，避免前端重复拼装规则

### 5.2 Analyst / Analytics 页面

这些页面分两类处理：

1. 已有真实聚合来源的页面：直接由 service + bridge + loader 回填首屏内容
2. 尚无对应业务模型的页面：改为展示当前系统真实可支撑的运营过程数据、分析快照、实验记录、任务统计、活动记录，或显示空状态

### 5.3 Dev/Test Seed 扩展

现有 `DevSeedService` 扩展为“页面查看级 seed”，除已有数据外，还应保证至少生成活动日志、分析快照、报表记录、工作流运行记录、实验项目与视图等真实记录。

## 6. 验证标准

1. `ui-notifications.js` 不再包含模拟通知定时注入逻辑
2. analyst/analytics 页面主展示内容由真实后端结果驱动或显示真实空态
3. 执行 seed 后，页面可看到真实数据库记录衍生出的内容
4. `G:/AI/TK-OPS-ASSISTANT-new/venv/Scripts/python.exe -m pytest tests/ -v` 通过
5. 启动桌面应用后，至少启动级 smoke test 正常
