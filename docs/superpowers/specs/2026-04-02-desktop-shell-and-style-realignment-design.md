# Desktop Shell 与样式回正设计

## 背景

当前新架构已经把数据链路与宿主链路基本打通，但开发入口默认落在浏览器 Vite 页，导致使用感知偏向“网页联调页”，而不是“桌面应用”。同时，新前端壳层样式没有延续旧项目的设计令牌和结构语义，造成视觉语言明显偏离旧系统。

这两个问题叠加后，会让用户看到一个“能跑，但不像原桌面应用”的界面。

## 设计目标

本轮纠偏明确做到三件事：

1. 默认开发入口回到 Tauri 桌面壳
2. 浏览器模式只保留为辅助调试入口
3. 新壳层和 Dashboard 首屏视觉语言回到旧项目体系

## 设计原则

### 1. 桌面壳优先

无论是开发、演示还是日常联调，默认入口都应是桌面宿主，而不是浏览器标签页。浏览器模式只能作为排障和前端单页调试手段存在。

### 2. 样式平迁而不是重画

本轮不重新发明第二套 UI。优先复用旧项目的：

- 设计令牌
- 壳层布局语义
- 按钮、标签、面板、表格的视觉语言
- 明暗主题基本规则

### 3. 数据链路不回退

视觉回正不能以牺牲新 runtime 数据链路为代价。页面仍然使用新前端与新 runtime 的接口，只是把展示层拉回旧设计系统。

## 入口设计

### `scripts/dev.ps1`

默认行为：

- 启动 Python runtime
- 启动 Vite dev server
- 启动 Tauri desktop host

浏览器模式：

- 保留为显式参数或独立脚本
- 不再作为默认开发命令

### `tauri.conf.json`

补齐开发态配置：

- `beforeDevCommand`
- `devUrl`
- `frontendDist`

让 Tauri 在开发模式下能正确载入 Vite 页面，而不是只能在构建后读取静态 `dist`

## 壳层设计

### 布局结构

新 AppShell 按旧系统语义维持：

- `title-bar`
- `sidebar`
- `main`
- `detail-panel`
- `status-bar`

对应 CSS 结构尽量贴近旧项目命名，避免再引入一套平行概念。

### 样式令牌

优先使用旧项目现有变量：

- `variables.css`
- `shell.css`
- `components.css`

必要时只做最小补充，不重新建立“米色玻璃态”那套视觉体系。

### 内容表达

TitleBar 和 StatusBar 应回到旧桌面工具的表达方式：

- 左侧品牌与路由信息
- 右侧宿主状态，但不喧宾夺主
- 少用长串调试串直出在主标题区域

## Dashboard 设计

Dashboard 首屏采用旧项目的工作台表达：

- 页面头：标题、副标题、顶部动作
- 统计卡：核心指标、状态分布、最近任务
- 面板：边界清晰、层次稳定
- 详情区：不再只是占位文案，而是辅助说明和系统摘要

保留新 runtime 的真实数据字段，但把文案与布局语言回收到旧项目风格。

## 风险与取舍

### 风险 1：旧 CSS 直接复用会和新组件类名不匹配

处理策略：

- 优先把新组件改成旧语义类名
- 不在本轮为此引入大规模样式映射层

### 风险 2：开发入口切回 Tauri 需要 dev 配置协同

处理策略：

- 优先补齐 `tauri.conf.json` 与脚本
- 若存在 CLI 依赖缺失，补齐到项目依赖中

### 风险 3：一次性全量视觉迁移范围过大

处理策略：

- 本轮只收口壳层与 Dashboard 首屏
- 其余页面后续按相同设计系统逐页跟进

## 完成标准

本轮完成后应满足：

- `scripts/dev.ps1` 默认打开桌面宿主而不是浏览器标签页
- 首屏看起来仍然是 TK-OPS 原有桌面产品，而不是另一套新网页
- 宿主状态信息存在，但不破坏主界面层级
- Dashboard 继续使用真实 runtime 数据
