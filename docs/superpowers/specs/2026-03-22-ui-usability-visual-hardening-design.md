# TK-OPS UI 可用性与视觉收口设计规格

## 1. 文档信息

- 日期：2026-03-22
- 主题：桌面壳层可用性修复、分析页信息表达重构、全局交互视觉统一
- 范围：只做补全、修复、收口，不新增业务功能
- 目标目录：`desktop_app/assets/css/`、`desktop_app/assets/js/`、必要的前端测试

## 2. 背景

当前主工作区已经完成了一轮样式基座恢复和第二轮密度收敛，但用户在真实运行中仍遇到一批阻塞使用的问题。问题不只发生在单页样式，而是同时涉及壳层布局、弹层层级、命中区、分析页组件表达、表单控件状态、分页缺失与按钮反馈不足。

用户已明确授权本轮按“壳层优先 + 页面联动”为主路线推进，同时要求页面逐个修复，并保证视觉观感同步提升。

## 3. 目标

本轮目标分为三层：

1. 先恢复稳定可用性：界面不遮挡、不漂移、不失焦、不出现点击穿透或 resize 后无法点击。
2. 再统一视觉语法：筛选、下拉、选中、按钮、分页、弹层、右下角说明区使用一致的组件语言。
3. 最后提升信息表达：尤其是分析页，图表与数据块必须能说明“发生了什么、问题在哪里、下一步做什么”。

## 4. 非目标

以下内容明确不在本轮范围内：

1. 不新增新的业务页面或新的业务能力。
2. 不引入新的前端框架、图表库或构建工具。
3. 不重做后端接口与数据模型，除非为现有页面正常渲染所必需。
4. 不优先处理未接线旧链路文件，如 `desktop_app/assets/routes/` 下的旧页面碎片。

## 5. 问题清单

### 5.1 全局壳层与弹层

1. 通知中心叠到下方区域，说明弹层层级、定位上下文或父容器裁切关系不稳定。
2. 顶部状态区域语义不清，“态”字表达抽象，用户无法快速判断其作用。
3. 菜单栏右下角说明区跟随滚动，导致用户必须滚到底部才能看到，违背持续状态区的用途。
4. 改变窗口大小后界面出现错位和不可点击，说明存在命中区与视觉区不同步、布局重排后遮罩未清、或 transformed 容器影响 pointer event 的问题。

### 5.2 页面与交互

1. 筛选项缺少完整样式，缺少分组、边界、选中反馈与节奏。
2. 下拉框样式不佳，且存在选中态异常。
3. 全局选中态不统一，有的过弱、有的混乱、有的几乎不可感知。
4. 某些按钮无法点击，或点击后没有任何交互反馈，用户无法判断是故障还是处理中。
5. 有的页面缺少分页，列表结尾像未完成态。

### 5.3 分析页

1. 图表看起来像装饰块，信息表达不清，用户无法从图中获得结论。
2. 电商转化页点击漏斗右侧出现大量弹框，属于交互绑定重复或弹层实例失控。
3. 粉丝画像页存在类似点击问题，说明分析页右侧联动面板和弹层机制未统一。
4. 其他分析页可能存在相同问题，需要按模式排查，而不是只修单个页面。

## 6. 设计原则

### 6.1 壳层先稳定，再谈页面体验

所有页面体验建立在稳定壳层之上。若滚动上下文、固定区、弹层层级和点击命中区不稳定，则局部视觉优化没有意义。本轮优先修复布局与交互基础设施。

### 6.2 一个组件，一个语法

筛选、下拉、标签、选中、分页、按钮、弹层必须在不同页面中使用同一组视觉与状态规则。页面可有局部差异，但不能再出现“同类控件像来自两个系统”的情况。

### 6.3 图表必须先服务结论

分析页不是为了展示复杂 UI，而是为了帮助用户快速判断趋势、异常与动作方向。图表与卡片必须带标题、时间/维度上下文、核心结论、图例或说明，弱化纯装饰性图形。

### 6.4 桌面窗口变化要可预测

响应式不追求移动端式折叠，而是要在桌面窗口变化时保持：布局不会炸、固定区不会失效、命中区不会漂移、弹层不会失控。

## 7. 方案总览

本轮采用“壳层优先 + 页面联动 + 组件统一 + 页面逐个清理”的组合方案。

实施顺序：

1. 全局壳层和弹层系统。
2. 全局组件状态统一。
3. 数据分析系页面逐页修复。
4. 列表/配置/表单页面补齐分页与反馈。
5. 针对 resize、点击、弹层等问题做跨页回归验证。

## 8. 全局壳层设计

### 8.1 布局与滚动上下文

壳层调整为“单一主滚动上下文 + 明确固定区域 + 受控辅助区”的结构：

1. `main` 作为主要滚动容器，负责页面主体滚动。
2. `title-bar`、`status-bar`、侧栏底部说明区保持固定语义，不与主内容竞争滚动。
3. `detail-panel` 在宽屏下保持独立辅助区；在窄窗口下应转为更明确的抽屉或堆叠，而不是挤压主内容造成命中区错乱。
4. 所有浮层都不得依赖会被 `overflow` 裁切的父容器作为最终展示边界。

### 8.2 弹层层级系统

建立明确的前端层级规则，至少区分：

1. 基础内容层。
2. Sticky 或固定工具层。
3. 下拉与轻量面板层。
4. 通知中心、搜索面板、右侧浮动详情层。
5. 模态或阻断型弹窗层。

所有通知中心、搜索面板、下拉菜单、分析侧片、快捷面板都必须使用统一层级策略，避免出现：叠在下层、被裁切、点击穿透、点一次生成多个弹层。

### 8.3 顶部状态区语义收口

顶部状态区保留，但从“抽象状态标签”改为“环境与运行摘要”。具体要求：

1. 避免只用“态”这样的单字标签。
2. 统一成用户能立刻理解的状态项，例如运行状态、连接状态、同步状态、告警数量。
3. 若信息过多，优先保留持续决策价值高的状态，弱化装饰性状态 pill。

### 8.4 侧栏底部说明区固定化

侧栏右下角说明区改为“侧栏底部冻结卡片”，始终贴底可见。该区域建议只保留高价值内容：

1. 当前版本或环境信息。
2. 当前连接 / 同步 / 更新摘要。
3. 一个最有价值的帮助提示、快捷键提示或入口。

不再放置大段说明文字，不作为文档说明区。

### 8.5 Resize 稳定性

针对窗口尺寸变化，要求：

1. 断点切换后布局立即重算，不允许视觉位置与真实点击区域错位。
2. 不允许存在残留遮罩层或透明层挡住界面。
3. `transform`、`pointer-events`、`position`、`overflow` 相关样式和脚本状态必须协同收口。
4. 若某些浮层在 resize 后锚点失效，必须自动关闭或重新定位。

## 9. 全局组件设计

### 9.1 筛选区

筛选区统一为一套“过滤条”组件，包含：

1. 条件标签与分组标题。
2. 输入框与下拉框。
3. 视图切换或排序控件。
4. 重置操作与结果统计。

视觉上应弱化碎片化按钮堆积，强调单一过滤区边界、分组节奏和已选条件反馈。

### 9.2 下拉框

下拉框统一要求：

1. 闭合态、展开态、悬停态、选中态、禁用态一致。
2. 选中项必须在视觉上足够明确，并与当前值同步。
3. 展开层不得被父容器裁切，不得在 resize 后漂移。
4. 键盘焦点与鼠标 hover 都要有可感知反馈。

### 9.3 选中态

选中态统一为“边框 + 背景 + 文本/图标强调”的组合语言，适用于：

1. 侧栏导航。
2. 页面 tab。
3. 筛选项。
4. 分析页图表维度切换。
5. persona 卡片、漏斗步骤、数据源卡片等可选择项。

### 9.4 按钮与反馈

所有按钮补齐：

1. `hover`
2. `active`
3. `focus-visible`
4. `disabled`
5. `loading` 或处理中反馈（若已有异步动作）

不能点的按钮必须看起来不能点；能点的按钮必须在点击后提供最基本反馈，不能像静态文本。

### 9.5 分页

所有有长列表或可分页语义的页面统一补齐分页控件，至少包含：

1. 当前页。
2. 上一页 / 下一页。
3. 总数或区间提示。
4. 若当前数据量不足分页，也保留统一的页脚信息结构，避免页面像未完成。

## 10. 数据分析页设计

### 10.1 信息结构重构

分析页统一使用以下信息顺序：

1. 页面标题与一句目标说明。
2. 筛选与时间/维度控制区。
3. 顶层关键指标或结论卡。
4. 主图表或主分析区。
5. 右侧洞察、建议动作、异常说明或明细区。

不再用纯视觉块堆出“分析感”。

### 10.2 图表表达规则

每个图表块必须至少包含：

1. 图表标题。
2. 数据时间范围或维度上下文。
3. 一句结论或用途说明。
4. 图例、标签或关键数值。
5. 当数据不足时的空状态说明。

图表视觉从“装饰图”收口为“低噪音信息图”：

1. 减少无意义的大面积渐变与花哨线条。
2. 强化标签、分段、对比、关键节点。
3. 让用户一眼知道高点、低点、异常点、转折点。

### 10.3 电商转化页

电商转化页的目标是先定位流失最大环节，再引导用户理解原因，因此：

1. 漏斗步骤可选中，但只允许单一激活态。
2. 右侧内容区应是稳定的详情面板或侧片，不允许点一个节点弹出多个面板。
3. 激活不同漏斗步骤时，只刷新同一块详情区域的内容。
4. 所有绑定必须避免重复注册，特别是在页面重复加载或切换后。

### 10.4 粉丝画像页

粉丝画像页应从“多个卡片可点”收口为“单选画像 + 辅助说明”：

1. 画像卡只能有一个当前聚焦对象。
2. 右侧联动信息保持单实例更新，不重复生成弹框。
3. 画像文案、标签、运营建议视觉上要分层，避免点了以后只看到噪音反馈。

### 10.5 分析系横向排查

除电商转化、粉丝画像外，还需按相同模式排查 `traffic-board`、`visual-lab`、`profit-analysis`、`competitor-monitor`、`report-center` 等页面，重点检查：

1. 是否存在重复事件绑定。
2. 是否存在浮层定位不稳定。
3. 是否存在选择态不清。
4. 是否存在右侧说明区信息弱、图表主体强装饰弱结论。

## 11. 列表 / 配置 / 表单页设计

### 11.1 列表与表格

1. 列表顶部保留统一筛选条。
2. 列表底部补齐分页或页脚摘要。
3. 选中行、hover 行、批量操作态统一视觉语言。
4. 长文本与窄列需要明确截断和提示，避免挤坏布局。

### 11.2 配置与表单

1. 表单输入、选择、切换、提示文本使用一致间距和高度。
2. 必填、错误、已保存、未保存状态必须清晰。
3. 下拉与选择类控件必须解决当前“样式不好看 + 选中错误”的问题。
4. 表单页面不再堆过多重复说明文案，保留必要 hint 即可。

## 12. 技术实施边界

### 12.1 优先修改的文件范围

1. `desktop_app/assets/css/variables.css`
2. `desktop_app/assets/css/shell.css`
3. `desktop_app/assets/css/components.css`
4. `desktop_app/assets/css/pages-analytics.css`
5. `desktop_app/assets/css/pages-core.css`
6. `desktop_app/assets/css/pages-config.css`
7. `desktop_app/assets/css/interactions.css`
8. `desktop_app/assets/js/ui-notifications.js`
9. `desktop_app/assets/js/charts-analytics.js`
10. `desktop_app/assets/js/page-loaders.js`
11. 其他真实运行链路上的前端 JS/CSS 文件

### 12.2 需重点排查的技术点

1. 重复绑定事件导致的多弹框问题。
2. `shell-hidden`、透明遮罩、绝对定位面板对点击的影响。
3. 由 `overflow`、`position`、`transform` 形成的新层叠上下文。
4. resize 后脚本未重新计算或状态未重置。
5. 组件在多页面中被不同样式覆盖造成的视觉不一致。

## 13. 验证策略

### 13.1 手动验证

至少验证以下场景：

1. 打开通知中心时不会被压到下层，也不会挡住错误区域。
2. 调整窗口宽度、最大化、恢复后，界面仍可正常点击。
3. 在分析页点击漏斗步骤或 persona 卡，不会生成多个重复弹框。
4. 筛选区、下拉框、选中态在不同页面中视觉一致。
5. 列表页与配置页的分页、按钮、空状态、禁用态可正常使用。

### 13.2 自动验证

根据现有测试能力，本轮至少补充前端基线测试，保护：

1. 关键壳层结构与核心组件选择器存在。
2. 关键分析页组件与分页组件选择器存在。
3. 若可行，补充针对重复绑定防回归的轻量测试或可验证脚本。

### 13.3 回归重点

1. 全局搜索、通知面板、详情侧栏之间不互相遮挡。
2. 分析页切换路由多次后，交互绑定不会倍增。
3. 窄窗口下的侧栏、详情区、浮层仍能维持正确位置和可点击性。

## 14. 成功标准

若本轮完成，用户应能感知到以下变化：

1. 主壳层稳定，不再出现遮挡、漂移和点击失效。
2. 筛选、下拉、按钮、选中、分页具有统一语言，看起来像同一套系统。
3. 分析页能看懂，不再只是“有图”，而是能读出结论与下一步动作。
4. 右下角说明区始终可见，内容更像运行摘要而不是噪音。
5. 页面逐个排查后，同类问题不会只在一个页面修好、在别处复发。

## 15. 实施顺序建议

1. 先修壳层与弹层系统。
2. 再修全局组件状态与统一语法。
3. 再逐页修分析系页面。
4. 再补列表 / 配置 / 表单页分页与反馈。
5. 最后做 resize、点击、弹层、选中态的跨页回归。

该顺序保证本轮先消除阻塞使用的问题，再保证视觉收口和页面级细节补完。
