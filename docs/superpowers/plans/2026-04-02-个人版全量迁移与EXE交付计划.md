# 个人版全量迁移与 EXE 交付计划（2026-04-02）

## 0. 背景与目标

> 适用范围：TK-OPS 当前仓库从“新架构已落主链路”进入“全量迁移 + 稳定性 + 可发布 EXE”阶段。

### 0.1 项目目标（本计划对齐）

1. 完全实现桌面应用，并最终交付给个人用户可安装、可运行的 EXE。
2. 当前产品定位收敛为个人用户，不引入团队协作复杂度。
3. 将旧代码能力全量迁移到新系统架构（以真实数据链路与真实交互为验收标准）。

### 0.2 当前基线

- 主链路：`apps/desktop`（Tauri + Vue3 + TS） + `apps/py-runtime`（Python runtime）已可运行。
- `desktop_app/` 仍存在可复用业务资产与迁移参考，需继续拆薄其对新链路的残余依赖。
- 当前阶段重点不是“页面能开”，而是“稳定、可恢复、可发布”。

---

## 1. 交付边界（In / Out）

### 1.1 In Scope（本轮必须达成）

- 旧系统核心业务能力在新架构完成迁移并可验证。
- 面向个人用户的一体化体验（安装、启动、设置、执行、导出、日志/错误反馈）。
- EXE 交付链可重复执行（构建、安装、基础回归、版本说明）。

### 1.2 Out of Scope（本轮明确不做）

- 团队组织管理、复杂权限体系、多人协作同步。
- 非核心炫技型新功能（与“可交付、可用、可维护”无关）。
- 无验证支撑的架构大重写。

---

## 2. 阶段计划（4 阶段）

## 阶段 A：盘点与冻结（范围治理）

### A.1 阶段目标

- 把“旧能力 -> 新架构落点 -> 当前状态 -> 缺口”一次性盘清。
- 形成个人版 MVP 白名单与弃用清单。

### A.2 核心任务

1. 建立全量迁移盘点表（页面、API、服务、数据模型、关键交互、导出能力）。
2. 标注每项状态：已迁移 / 半迁移 / 未迁移 / 可废弃。
3. 冻结个人版目标清单（P0/P1/P2）。

### A.3 文件地图（预计）

- `docs/migration/page-matrix.md`
- `docs/migration/current-status-and-roadmap.md`
- `docs/superpowers/plans/*`（新增阶段状态文档）

### A.4 验证方式

- 盘点表覆盖率 = 100%（无“未分类能力”）。
- 每项能力有明确负责人和下一动作。

### A.5 回退点

- 若范围超载：回到 P0 白名单，只保留个人用户核心链路。

---

## 阶段 B：全量迁移实现（能力闭环）

### B.1 阶段目标

- 完成未迁移/半迁移能力的前后端真实打通。
- 消除“假数据可演示、真实场景不可用”的断层。

### B.2 核心任务

1. 前端：
   - 迁移页面运行时接线（loader/bindings/data 调用统一化）。
   - 补齐异常提示、空态、可恢复交互。
2. 后端：
   - 补齐 runtime API 与 bridge/service/repository 缺口。
   - 完成数据模型与迁移脚本兼容。
3. 合同：
   - 统一错误 envelope 与事件通知，避免前端分支散落。

### B.3 文件地图（预计）

- `apps/desktop/**`
- `apps/py-runtime/src/**`
- `desktop_app/services/**`（仅复用/迁移来源）
- `desktop_app/database/migrations/**`
- `tests/**`

### B.4 验证方式

- 改动相关测试通过（契约 + 交互 + 服务层）。
- 至少 1 条主链路回归通过（从启动到完成关键动作）。

### B.5 回退点

- 发生高风险回归时，按域回退到前一稳定 tag/commit，不跨域连锁回退。

---

## 阶段 C：稳定性与个人版体验收口

### C.1 阶段目标

- 提升 sidecar 生命周期稳定性与故障可恢复性。
- 用个人用户视角优化关键流程文案和反馈。

### C.2 核心任务

1. sidecar 启停、心跳、异常退出恢复、超时处理。
2. 关键失败路径统一文案与引导（重试/修复建议/日志定位）。
3. 数据目录与历史数据库兼容性验证。

### C.3 文件地图（预计）

- `apps/desktop/src-tauri/**`
- `apps/py-runtime/src/bootstrap/**`
- `apps/py-runtime/src/api/**`
- `tests/test_startup_*`、`tests/test_runtime_*`、`tests/test_release_*`

### C.4 验证方式

- 启动/重启/故障恢复 smoke 通过。
- 关键错误路径具备可见反馈与可恢复动作。

### C.5 回退点

- sidecar 稳定性出现阻塞时，启用降级启动策略并保留最小可用功能。

---

## 阶段 D：EXE 交付与发布治理

### D.1 阶段目标

- 产出可分发 EXE，并完成安装-启动-回归-发布说明闭环。

### D.2 核心任务

1. 构建：固定发布命令、版本号、产物结构。
2. 验收：安装后首启、关键主链路、升级覆盖回归。
3. 发布：发布记录、已知问题、回滚说明。

### D.3 文件地图（预计）

- `scripts/release.ps1`
- `scripts/build-desktop.ps1`
- `installer.iss`
- `docs/releases/*.md`
- `README.md`

### D.4 验证方式

- EXE 安装包在目标 Windows 环境可完成首启和核心链路。
- 发布记录与版本信息一致。

### D.5 回退点

- 发布阻塞时回退到上一个可用版本并发布 hotfix 说明。

## 4. 风险清单与对策

1. **迁移范围蔓延**：
   - 对策：P0 白名单冻结，新增需求进入下一迭代池。
2. **旧壳隐性依赖难拆**：
   - 对策：建立依赖清单，按域替换，逐步断链。
3. **打包后与开发环境行为不一致**：
   - 对策：发布前必须执行安装后 smoke 与数据目录兼容回归。
4. **异常路径不可见导致“看似成功”**：
   - 对策：统一错误信封 + UI 提示 + 日志追踪 ID。

---

## 5. 阶段验收门禁（必须全部满足）

- 功能门禁：核心能力真实可用（非假数据演示）。
- 质量门禁：关键自动化测试通过 + 主链路 smoke 通过。
- 发布门禁：EXE 可安装、可启动、可执行关键动作。

## 7. 基于 4 月 1 日文档的深化补充（执行硬约束）

为避免“计划有了、落地走形”，本计划补充以下硬约束，作为实施期日常检查项：

### 7.1 单壳硬约束（必须）

1. Tauri 是唯一允许继续演进与发布的桌面宿主。
2. `desktop_app/` 仅作为迁移参考与业务资产复用来源，不进入默认联调与发布路径。
3. 禁止“旧壳跑业务、新壳做界面”的双轨开发模式。

### 7.2 小文件硬约束（必须）

1. 对核心代码启用小文件警戒线，避免再形成桥接 God Object 与巨型 loader。
2. 新增页面与服务必须按职责拆分，不得用注释分区替代结构化拆分。
3. 超过警戒阈值的文件必须在当迭代内拆分，不得挂账。

### 7.3 协议边界硬约束（必须）

1. runtime `session_token` 作为优先鉴权来源，HTTP / WS 保持同口径握手语义。
2. `/health` 除可探活外，必须承载最小协议边界元信息，支撑前端可靠探测。
3. 协议文档与实现同步更新，禁止“代码已改、文档未收口”。

### 7.4 阶段性 KPI（新增）

- 阶段 A 结束：能力盘点覆盖率 100%，P0/P1/P2 冻结并完成负责人绑定。
- 阶段 B 结束：P0 能力迁移完成率 >= 95%，且无阻塞发布的 P0 缺陷。
- 阶段 C 结束：启动/恢复类 smoke 连续通过，关键异常路径具备可恢复引导。
- 阶段 D 结束：安装后回归通过并产出发布记录，可回滚链路可演练。

### 7.5 每周例行审查清单（新增）

1. 是否出现向 `desktop_app/` 新增业务能力的行为。
2. 是否新增或继续放大超大文件。
3. 是否存在 HTTP/WS 协议不一致与前后端字段漂移。
4. 是否存在“仅开发环境可用、打包后不可用”的能力。
