# TK-OPS 当前仓库重构方案：项目脚手架清单 + 详细任务清单 + 目标

## 1. 当前仓库现状判断

当前仓库并不是空壳，而是已经形成了一套可运行的本地桌面方案：

- 使用 **PySide6 + QtWebEngine** 承载桌面窗口与本地 Web Shell。  
- Python 侧通过 **QWebChannel Bridge** 暴露服务给前端。  
- 数据层已经有 **SQLite + SQLAlchemy + Alembic**。  
- 已经存在较完整的 service 层、seed、sample db、自动化测试。  
- 关键页面已经开始接真实后端数据，而不是纯静态原型。

从 README 看，当前架构已经明确是：

- Python 负责桌面外壳、数据库、服务层、bridge 与系统能力  
- PySide6 + QtWebEngine 承载桌面窗口  
- HTML / CSS / JavaScript 负责页面结构、交互和运行时渲染  
- Python 与前端通过 QWebChannel 双向通信  
- SQLite + SQLAlchemy + Alembic 负责本地持久化

这说明项目已经完成了“从纯原型到真实后端接线”的第一阶段，不适合推倒重来，适合做**渐进式重构**。

---

## 2. 当前仓库的核心问题

### 2.1 优点
1. 已经是本地桌面应用，打包链路存在。
2. 数据库、迁移、样本库、测试基线已经存在。
3. 账号、任务、AI 供应商、素材、分析、报表、工作流已经有真实数据路径。
4. 开发 seed 和 sample_data 对联调很有帮助。
5. 说明项目已经积累了不少业务建模成果，不该全部废弃。

### 2.2 主要问题
1. **桌面壳与业务绑定过深**  
   `desktop_app/app.py` 直接启动 `WebShellWindow`，桌面启动、数据库预热、窗口展示高度耦合。

2. **Bridge 过于巨大，已经接近 God Object**  
   `desktop_app/ui/bridge.py` 同时承担：
   - 服务实例化
   - slot 暴露
   - 文件选择
   - 流式 AI
   - License
   - 更新
   - 网络诊断
   - clipboard
   - logs  
   这会导致后续迁移 Tauri/Vue 时非常难拆。

3. **前端页面层太重，且偏原生 DOM imperative**
   `routes.js` 和 `page-loaders.js` 已经非常巨大，尤其 `page-loaders.js` 既有运行时摘要、又有页面加载器、又有大量 DOM 操作和页面逻辑，长期维护成本高。

4. **前端技术栈不适合继续扩复杂交互**
   当前是 HTML/CSS/JS，不是 Vue/TS。  
   对你后续要做的：
   - 视觉编辑器
   - AI 内容工厂
   - 可视化实验室
   - 更复杂的 Dashboard
   来说，维护难度会持续上升。

5. **Repository 还是“通用仓库 + 专用方法”混合模式**
   `repository.py` 还不是按聚合边界拆分的仓储体系，而是一个大仓库包很多模型和计数逻辑，后续会继续膨胀。

6. **进程模型还停留在“单桌面进程 + bridge”心智**
   这对“AI 深度接入 + 长任务 + 流式输出 + 调度器”来说，后期很容易变成稳定性瓶颈。

---

## 3. 重构目标

这次重构的目标不是“换技术栈”，而是：

### 总目标
把当前仓库从：

**PySide6 + QtWebEngine + QWebChannel + 原生 JS 页面**

重构为：

**Tauri + Vue 3 + TypeScript + Python Sidecar Runtime + SQLite**

### 目标拆解
1. **保留已有 Python 业务资产**
   - models
   - migrations
   - services
   - dev seed
   - sample db
   - 现有测试中可复用的部分

2. **替换桌面宿主层**
   - 从 PySide6/QWebEngine 切换到 Tauri

3. **替换前端表示层**
   - 从原生 HTML/CSS/JS 切换到 Vue 3 + TypeScript

4. **重做通信边界**
   - 用 Tauri command 处理短调用
   - 用 Python WebSocket 处理长任务和流式输出
   - 不再依赖 QWebChannel

5. **把 Python 提升为 runtime**
   - 不再是“桌面程序内部服务模块”
   - 而是“独立 sidecar 运行时”

6. **先稳定 P0 主链路，再迁复杂页面**
   - 不是一次性把所有页面全部重写

## 6. 当前代码保留与迁移建议

## 6.1 直接保留
这些资产不建议重写：

1. `desktop_app/database/models.py`
2. `desktop_app/database/migrations/`
3. `sample_data/`
4. `tests/` 中与数据库、service 行为、seed、analytics 聚合相关的测试
5. `desktop_app/services/` 中真正有业务价值的 service

### 保留目标
- 减少业务逻辑重写成本
- 保住现有数据模型积累
- 保住已有测试收益

---

## 6.2 需要拆分迁移
这些不能直接搬过去，要先拆：

1. `desktop_app/ui/bridge.py`
2. `desktop_app/assets/js/routes.js`
3. `desktop_app/assets/js/page-loaders.js`
4. `desktop_app/ui/web_shell.py`
5. `desktop_app/app.py`

### 拆分目标
- 让桌面壳、UI、业务、运行时边界清晰
- 避免 God Object
- 避免 giant JS loader file 继续增长

---

## 6.3 可以废弃
这些在新架构中最终会退出：

1. QWebChannel 绑定逻辑
2. QtWebEngine 前端承载逻辑
3. 原生 JS 页面工厂与模板注入体系
4. 旧的 bridge slot 暴露方式

## 10. 最终建议

### 不建议
1. 直接在当前原生 JS 页面上继续堆功能
2. 一次性全量重写所有 27 个页面
3. Python runtime 和前端继续共处一个 bridge 巨类
4. 迁移时把数据库层也一起大改

### 建议
1. **先抽 runtime，再换壳，再迁页面**
2. **先迁 P0 页面，再迁 P1/P2**
3. **保住数据模型、迁移、service、seed、测试资产**
4. **用 Vue/TS 接管复杂交互页**
5. **用 Tauri 管桌面能力，用 Python 管业务与 AI**

---

## 11. 一句话结论

当前项目最正确的重构方式不是推倒重来，而是：

**保留 Python 业务核心和数据资产，淘汰 PySide6 + QWebChannel + 原生 JS 壳层，迁移到 Tauri + Vue3 + TypeScript + Python Sidecar Runtime。**

这条路线可以最大限度保住已有成果，同时把你最关心的：

- exe 交付
- 本地运行
- 界面美观
- 复杂交互
- AI 深度接入

全部统一起来。
