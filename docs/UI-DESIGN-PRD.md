## 2. 文档目标

本 PRD 用于统一 TK-OPS 桌面应用的视觉语言、交互框架、组件标准、主题规范与页面落地方式。文档面向产品、设计、前端桌面开发、测试与后续扩展团队，确保以下内容在实现阶段保持一致：

1. 全局外壳结构一致。
2. 所有页面在同一设计系统内运行。
3. 所有颜色、字体、间距、圆角、阴影、动画均可映射为可执行 QSS 令牌。
4. 26 个页面的布局、组件、交互、局部导航与右侧详情面板行为有明确标准。
5. Light / Dark 双主题可以在不改动页面结构的前提下切换。
6. 后续新增页面可直接复用现有壳层、导航、组件与设计令牌。

## 3. 产品与界面定位

TK-OPS 是一个面向 TikTok Shop 运营团队的企业级桌面工作台。它不是单一工具，而是一个跨角色、多任务、强状态、强流程的复合型操作系统。UI 的核心职责不是做出营销感，而是让用户在高密度信息下依然能快速定位、判断、执行、回看。

### 3.1 角色覆盖

1. 管理者，关注总览、风险、效率、调度。
2. 内容创作者，关注素材、脚本、批量生成、编辑工作流。
3. 数据分析师，关注趋势、竞品、利润、互动、报表。
4. 自动化运营人员，关注任务、发布、回复、采集与规则运行状态。
5. 系统维护人员，关注配置、网络、传输、下载、日志、升级。
6. CRM 运营人员，关注客户分层、触达、跟进、标签与摘要。

### 3.2 产品气质

1. 专业，不炫技。
2. 信息密度高，但阅读压力低。
3. 交互明确，不依赖猜测。
4. 多任务并行，可随时切换上下文。
5. AI 能力可见，但不喧宾夺主。
6. 工具型页面与工作台型页面能共存于统一壳层。

## 4. 设计理念

### 4.1 核心原则

#### 4.1.1 壳层稳定，业务变化

用户在任何业务页都应感知同一套外壳结构。导航位置、搜索入口、通知入口、状态信息、详情面板开合方式保持稳定，降低跨模块切换成本。

#### 4.1.2 信息分层清晰

每个页面都按四层信息组织：

1. 全局层，Title Bar、Sidebar、Status Bar。
2. 页面层，Breadcrumb、标题、局部导航、主操作条。
3. 内容层，卡片、表格、图表、画布、表单、时间轴。
4. 辅助层，Detail Panel、悬浮操作、批量操作条、告警条。

#### 4.1.3 先判断，再操作

所有高风险动作必须有前置状态提示。比如发布、删除、批量运行、覆盖配置、重试异常任务，都必须先让用户看见影响范围、对象数量、当前状态。

#### 4.1.4 AI 是增强器，不是主入口

AI 功能应嵌入业务流程，如标题优化、脚本改写、合规检测、调研报告、卖点生成、智能增强，而不是独立存在成无法理解的黑盒。

#### 4.1.5 桌面优先

本产品以桌面高效操作为第一原则。支持宽屏、多栏、高密度表格、细粒度筛选、键盘快捷与持续状态栏，响应式策略服务于桌面窗口变化，不追求移动端式单栏折叠体验。

### 5.3 壳层区域定义

#### 5.3.1 Title Bar

职责：

1. 应用身份展示。
2. 当前模块标题反馈。
3. 全局搜索。
4. 通知与系统消息。
5. 用户与环境入口。

交互要求：

1. 支持拖动窗口。
2. 全局搜索支持页面跳转、实体定位、最近访问。
3. 通知中心支持未读状态与清空操作。
4. 用户头像支持个人设置、退出、版本信息入口。

#### 5.3.2 Sidebar

职责：

1. 一级主导航。
2. 角色分组识别。
3. 当前页面定位。
4. 折叠模式下的快速切换。

交互要求：

1. 展开宽度 280px。
2. 折叠宽度 64px。
3. 折叠后仅保留图标轨道与 Tooltip。
4. 激活项使用品牌色高亮底与左侧强调条。
5. 组标题在展开模式下显示，在折叠模式下隐藏。

#### 5.3.3 Content Host

职责：

1. 作为 `QStackedWidget` 主页面容器。
2. 承载页面顶部工具条、内容主体、页内导航、分页、视图切换。
3. 承载主滚动上下文。

#### 5.3.4 Detail Panel

职责：

1. 展示所选对象的上下文详情。
2. 承载次级配置。
3. 承载实时预览、属性检查、合规报告、节点配置、日志片段。

规范：

1. 默认宽度建议 360px，允许 320px 到 420px 区间。
2. 不作为页面主内容的唯一入口。
3. 可关闭，可切换 Tab，可保持选择状态。
4. 当窗口宽度不足时可转为抽屉式浮层。

#### 5.3.5 Status Bar

职责：

1. 展示连接、同步、校验、任务、运行状态。
2. 对后台任务型页面提供持续反馈。
3. 在 AI 工作流、下载、网络诊断、任务队列等页面承载重要运行信息。

## 7. 主题系统

### 7.1 主题目标

1. Light 模式适合白天办公、报表阅读、批量管理。
2. Dark 模式适合长时间使用、视频编辑、图表分析、夜间运营。
3. 主题切换不改变布局，只改变语义色映射、边框强度、阴影密度、图表默认配色排序。

### 7.2 Light 模式定义

1. 页面主背景使用 `surface.primary = #F5F8F8`。
2. 卡片与面板使用 `surface.secondary = #FFFFFF`。
3. 表头与次级底使用 `surface.tertiary = #F8FAFC`。
4. 表单与下沉区使用 `surface.sunken = #F1F5F9`。
5. 主文字使用 `text.primary = #0F172A`。
6. 辅助文字使用 `text.secondary = #64748B`。

### 7.3 Dark 模式定义

1. 页面主背景使用 `surface.primary = #0F2323`。
2. 卡片与面板使用 `surface.secondary = #0F172A`。
3. 次级底与表头使用 `surface.tertiary = #1E293B`。
4. 输入与下沉区使用 `surface.sunken = #1E293B`。
5. 主文字使用 `text.primary = #F1F5F9`。
6. 辅助文字使用 `text.secondary = #94A3B8`。

### 7.4 主题切换行为

1. 切换时不刷新页面数据。
2. 切换时保持滚动位置、选中项、局部 Tab 状态。
3. 图表切换时需同步更新坐标轴、网格线、Tooltip 容器背景。
4. 视频编辑与工作流画布页面需优先保证对比度，不允许因主题切换导致时间轴、连线、节点标签不可读。

## 8. 布局规范

### 8.1 页面宽度策略

1. 默认内容区最大化铺满 Content Host。
2. 页面左右内边距使用 `32px`。
3. 一级模块之间垂直间距使用 `24px`。
4. 卡片内边距统一 `24px`。
5. 栅格优先采用 12 列心智模型，但在 Qt 实现中可使用弹性布局与固定宽面板组合。

### 8.2 典型布局模板

#### 模板 A，数据看板页

上层为 KPI 卡片，中层为图表，下层为列表或活动流。

#### 模板 B，管理列表页

顶部筛选与操作条，中部主表格或卡片列表，右侧详情面板。

#### 模板 C，创作工作台页

左工具区，中主画布，底时间轴或结果区，右属性面板。

#### 模板 D，配置中心页

左本地导航，中配置表单，右说明或实时校验面板。

#### 模板 E，任务运营页

顶部状态过滤，中任务列表或日历，底部状态栏。

### 8.3 Shell 区域尺寸建议

| 区域 | 建议尺寸 | 说明 |
|---|---|---|
| Title Bar | 56px 高 | 全局搜索与用户操作 |
| Sidebar 展开 | 280px 宽 | 主导航 |
| Sidebar 折叠 | 64px 宽 | 图标轨道 |

### 8.4 表单布局规范

1. 双列表单优先，标签在上，输入在下。
2. 高风险配置必须包含说明文本、当前状态与测试按钮。
3. 密集型设置页支持左侧二级导航。

1. 同一屏不超过 4 个主图表。
2. 图表上方必须有标题、时间范围、指标切换。
3. 图表下方或右侧需提供摘要卡，不让用户仅凭视觉曲线判断。
4. Dark 模式下网格线透明度降低，强调数据线与 Tooltip。

## 12. 动画规范

### 12.1 动画目标

动画只服务反馈、层级切换与状态确认，不做无意义装饰。

### 12.2 交互场景与建议时长

| 场景 | 时长 | 曲线 |
|---|---|---|
| 按钮 Hover | `150ms` | `easing.default` |
| Tab 切换 | `150ms` 到 `250ms` | `easing.default` |
| Detail Panel 展开 | `250ms` | `easing.default` |
| Sidebar 折叠 / 展开 | `250ms` | `easing.default` |
| 模态弹入 | `250ms` | `easing.default` |
| 批量结果完成高亮 | `400ms` | `easing.bounce` |
| 加载骨架过渡 | `400ms` | `easing.default` |

### 12.3 动画使用规范

1. 表格排序只允许轻微透明度与位置过渡。
2. 图表刷新优先使用数据更新动画，不整体闪烁。
3. 时间轴拖动不使用弹跳动画。
4. 网络诊断、下载、队列页面允许使用线性进度和转圈，但要能中断。

## 13. 响应式策略

### 13.1 响应范围

本产品为桌面应用，响应式仅针对窗口缩放与常见办公显示器宽度，不面向移动端。

### 13.2 断点建议

| 窗口宽度 | 模式 | 策略 |
|---|---|---|
| `>= 1600px` | 宽屏工作站 | 保持 Sidebar 展开，允许 Detail Panel 常驻，多列卡片和双图表并排 |
| `1366px - 1599px` | 标准桌面 | 默认 Sidebar 展开，Detail Panel 按需展开 |
| `1200px - 1365px` | 紧凑桌面 | Sidebar 可自动折叠，主内容减少一列 |
| `960px - 1199px` | 最小支持 | Detail Panel 改抽屉，局部工具栏折行为更多菜单 |
| `< 960px` | 不推荐 | 提示用户扩大窗口，保留核心表单与列表 |

### 13.3 响应动作

1. Sidebar 先折叠，再压缩内容。
2. Detail Panel 优先转抽屉，不挤压主内容。
3. KPI 卡从四列变两列，再变单列。
4. 图表从并排改上下堆叠。
5. 表格工具栏的次级按钮折叠到更多菜单。

## 14. 无障碍设计

### 14.1 对比度

1. 正文与背景对比度不低于 4.5:1。
2. 大字号标题不低于 3:1。
3. 品牌色按钮上的文字固定使用深色 `#0F2323`，保证可读性。

### 14.2 键盘可达性

1. Sidebar、Tab、表单、表格操作需支持键盘焦点遍历。
2. 焦点环统一使用 `border.focus`。
3. 关键操作支持快捷键，如搜索、保存、运行、导出、上一页下一页。

### 14.3 语义反馈

1. 不可只靠颜色表达成功、错误、在线、离线。
2. 所有状态需同时提供文本或图标说明。
3. 表单错误需贴近字段展示，不只在顶部汇总。

### 14.4 动效友好

1. 高速闪烁禁止。
2. 长时间运行页的加载动效要低干扰。
3. 支持减少动画模式时，保留状态切换但降低位移动画。

## 17. 统一交付要求

1. 所有页面必须至少覆盖加载中、空状态、正常状态、错误状态四类反馈。
2. 页面标题、按钮、状态文案统一使用中文业务语义，不使用模糊词。
3. 所有实现优先接入语义令牌与 QSS 映射，不写死颜色、圆角、阴影。
4. MainWindow 负责 Shell，`QStackedWidget` 负责 Content Host，Detail Panel 走统一容器。
5. 验收时重点检查 Sidebar 折叠、主题切换、Detail Panel 可用性、表格图表层级、AI 上下文清晰度。

## 18. 结论

本 PRD 已定义 TK-OPS 桌面应用的统一 UI 体系，覆盖设计理念、设计系统、布局规范、组件库、主题系统、导航设计、图标规范、动画规范、响应式策略、无障碍设计，以及 26 个页面的完整布局与交互说明。后续实现应先完成 Shell 与基础组件，再按页面规格逐步落地业务能力。
